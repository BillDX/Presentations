> CLAUDE.md

CLAUDE.md — The Brief on Day One

A markdown file. Auto-loaded at the start of every Claude Code session. The closest thing the agent has to onboarding.

Where it lives (merged in order)

  • ~/.claude/CLAUDE.md — your global voice
  • project/CLAUDE.md — the team's brief (committed)
  • project/<subdir>/CLAUDE.md — loaded when working there

Rule of thumb

Things you'd tell a new contractor on day one. Conventions, gotchas, "don't touch X".

KEEP IT SHORT Every line costs context. Aim for < 150 lines.

# project/CLAUDE.md

## Project
FastAPI service for conference
talk submissions. Postgres backend.

## Conventions
- We use uv, not pip
- Tests live in tests/, never inline
- Type hints on every public fn

## Don't
- Touch migrations/ without sign-off
- Run pytest -k in CI mode

## Helpful
- Local DB: make db.up
- Smoke tests: make smoke

## Docs
- For UI/UX work refer to /docs/styleguide.md
- For testing guidelines refer to /docs/coding_standards.md
> .claude/agents/

Agent Definitions — Reusable Specialists

A markdown file per agent in .claude/agents/. Frontmatter declares capabilities; the body is the system prompt. Ship them in your repo so the team uses the same specialists you do.

# .claude/agents/reviewer.md
---
name: reviewer
description: Reviews code for
bugs, security, and style.
tools: Read, Grep, Glob
model: sonnet
isolation: worktree
---
You are a code reviewer. Read the
specified files and report issues
grouped by severity.

Key Levers

  • tools: restrict capability surface (read-only? write? Bash?)
  • model: route to opus / sonnet / haiku by job
  • isolation: worktree sandboxes the filesystem
  • description: Claude reads this to decide when to invoke

Library mindset: a small set of well-named, well-scoped agents beats a giant generic one. Build them like you build internal tools.

> commands + hooks

Slash Commands & Hooks

Two ways to encode workflow into the harness: slash commands for repeatable prompts, hooks for programmatic gates around tool calls.

Slash Commands

A markdown file becomes a /command. Saves typing the same prompt twice.

# .claude/commands/review.md
---
description: Run the reviewer agent
on the current branch's diff.
---
Use the reviewer agent to review
all changes in the current branch
vs main. Group findings by file.

In session: /review

Hooks

Shell commands fired on events: PreToolUse, PostToolUse, Stop, SessionStart.

# .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "command": "ruff format $FILE"
      }]
    }]
  }
}

Effect: auto-format on every edit, no prompt.

> settings.json

Permissions — The Trust Budget

Every permission prompt mid-flow is friction. settings.json lets you decide once which calls are auto-approved and which are denied outright.

You Build It By Using Claude

Most of your allow list writes itself. When Claude asks for permission, you'll see:

Bash command: uv run pytest
[a] Allow once
[A] Always allow ← writes to settings.json
[d] Don't allow

Pick "always allow" — the rule lands in .claude/settings.json. Next time, no prompt.

ALSO Edit settings.json by hand to seed the allowlist before you start, or to add deny rules — those are click-once-never.

# .claude/settings.json — grown by clicking
{
  "permissions": {
    "allow": [
      "Read", "Edit", "Glob",
      "Bash(git status:*)",
      "Bash(uv run pytest:*)",
      "Bash(make:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Write(./.env)",
      "Write(./.env.*)"
    ]
  }
}

TIP The /fewer-permission-prompts skill scans your transcripts for repeated approvals and proposes additions in bulk.

> .claude/skills/

Skills — Procedures Claude Loads on Demand

A skill is a packaged procedure: a folder with a SKILL.md describing when to use it and how. Claude reads the descriptions of every available skill and loads the relevant one automatically when your task matches.

# .claude/skills/release-notes/SKILL.md
---
name: release-notes
description: Generate release notes
  for a tag range. Use when the user
  asks "what shipped" or for a changelog.
---

## Process
1. Run scripts/collect-prs.sh
2. Group by area using AREAS.md
3. Output in TEMPLATE.md format

A skill folder ships scripts, templates, examples — whatever the procedure needs. The description is how Claude knows to reach for it.

Skills vs Slash Commands

  • /command — explicit. You type it.
  • Skill — auto-discovered. Claude picks it up when the description matches your ask.

Rule of thumb: cross-cutting procedure that should fire whenever it applies? Skill. One-off prompt shortcut you want to type yourself? Slash command.

> sub-agents

Sub-Agents — The First Move Up

The simplest way out of the single-agent ceiling. The parent spawns specialized workers, each with its own context window. Hub-and-spoke, not mesh.

Their job: serve your current context window. They explore, summarize, or do a small piece of work — then hand the answer back.

MAIN SESSION (YOU)

orchestrates + receives results

|--------+--------+--------|
[REVIEWER]

Read-only critique

[DOCUMENTER]

Adds docstrings

[EXPLORE]

Locates code

`-- results --> parent <-- results --'

KEY Sub-agents cannot message each other. The parent is the only hub.

What You Get

  • Each agent gets a fresh context window — no crosstalk
  • Tool restrictions enforce blast radius
  • Cheaper agents (haiku) for grunt work; opus for thinking
  • Parallel by default — faster than sequential
USE FOR

Bounded tasks directly tied to what you're working on. Review this file. Document these handlers. Find usages of X.

DON'T USE FOR

Open-ended exploration. Whole-feature builds. Anything you can't describe in one sentence with a clear stop condition.

What Just Happened

The Wins

  • Two fresh context windows — no crosstalk

The Takeaway

You have two ways to spin up a sub-agent:

ON THE FLY

Like we just did. Describe the agent in the prompt — tools, model, job — and Claude spins it up for that turn. Disposable, fast, zero setup.

PERSISTED IN .claude/agents/<name>.md

Permanent, reusable, committed. Becomes part of the repo. The whole team gets the same specialists, the same way you do. Build a library.

> ~/containment

Containment Lives in the Agent Definition

Demo 1's reviewer.md and documenter.md were already containing the agent — through three frontmatter fields. Each one bounds a different kind of risk.

# .claude/agents/reviewer.md
---
name: reviewer
description: read-only critique
tools: Read, Grep, Glob  # ←
model: sonnet              # ←
isolation: worktree         # ←
---

Three fields. Three safety rails. Use them to keep your agents focused, bounded, and on task — not wandering into parts of the codebase they have no business touching.

tools: — capability surface

Whitelist of what the agent can do. Read-only? Edit but no Bash? Closes off whole categories of failure before the agent runs.

model: — cost & horsepower

Match the model to the work. Haiku for grunt work, sonnet for code, opus for planning. Cheaper agents are also a containment lever — less rope to hang with.

isolation: — filesystem blast radius

Where the agent's writes land. worktree means a separate checkout — mistakes stay on a throwaway branch, not your real tree.

The next three slides go deeper on the third one — worktrees, sandboxed bash, and containers — because that's where the most damage usually lives.

> git-worktree

Git Worktrees — The Cheap, Boring Primitive

A worktree is a second checkout of the same repo on a different branch, in a separate directory. Built into git. Free. Most people forget it exists.

Why It Works for Agents

  • Each agent gets its own filesystem copy
  • No file collisions when two agents edit at once
  • Results merge back via git — you review the diff
  • If an agent breaks something, only its worktree dies
  • Claude Code can manage worktrees automatically — isolation: worktree
# worktree = separate checkout
$ git worktree add ../proj-coder coder-branch
$ git worktree list
/project abc1234 [main]
/project-coder def5678 [coder-branch]
/project-tester ghi9012 [tester-branch]

# cleanup when done
$ git worktree remove ../proj-coder

RULE If two agents touch the same repo at the same time, they need different worktrees. Full stop.

> ~/security-aside

A Quick Word on Agentic Security

Containment is local hygiene. There's a larger conversation about agentic and LLM-powered apps that's out of scope today — but you should know it exists before you ship anything that lets an agent touch the outside world.

The Lethal Trifecta

Simon Willison's framing for what makes an agent dangerous:

1. Access to private data

Your repo. Your secrets. Your customers' info.

2. Exposure to untrusted content

A web page, an issue, an email body, a PDF.

3. Ability to communicate externally

Send mail, hit an API, write a file someone else can read.

All three? An attacker can trick the agent into exfiltrating your data through the agent's own permissions.

What This Means for You

  • Coding agents on a private repo with no internet? Lower risk.
  • An agent that browses the web and can edit your codebase? Trifecta — treat it like prod.
  • The prompt is code. Untrusted prompts are untrusted code.

Out of scope today. If you're shipping agentic apps to users, the deeper version of this material lives in "Securing Your Agents" — same Presentations repo.

> --dangerously-skip-permissions

When --dangerously-skip-permissions Is OK

The flag does what it says. It is wildly useful. It is also how you wake up to a deleted branch, a broken database, or a $300 token bill. Use it — but only with a net.

OK to Use When…

  • The agent is in a container or VM
  • The agent is in a worktree on a throwaway branch
  • You have deny rules in settings.json for the truly dangerous calls
  • You're watching — or have a stop hook
  • Token spend is bounded by a quota or budget alert

Not OK When…

  • You're on your primary working tree
  • The agent has access to prod credentials or your real shell
  • The agent can push, deploy, or call paid APIs without limit
  • You're going to bed

Mental model: the flag isn't "skip safety." It's "I've moved the safety to a different layer." If you haven't, don't pull it.

> controller-pattern

The Controller Pattern

The same three-move shape, scaled up. Plan → Execute → Review — with you as the controller, and as much (or as little) parallelism in the middle as the work needs.

The Shape

CONTROLLER (YOU)

owns the plan, merges the work

|--- routes work ---|
1. PLAN

PLAN.md, approved

2. EXECUTE

1 agent or N, in worktrees

3. REVIEW

REVIEW.md vs the plan

`-- artifacts you can read --'

KEY The plan and the review are documents. They survive the conversation. They're reviewable, editable, and version-controllable.

Why It Wins

  • PLAN.md — you steer before tokens get burned on the wrong path
  • Execute step scales: one agent for small work, sub-agents in parallel worktrees for big work
  • REVIEW.md — Claude is held to its own plan. Deviations get called out, not buried
  • Failure stays scoped — you find it in writing, before merge

When to reach for it: any task where you'd normally regret what Claude did before you could intervene. Feature-from-spec, multi-file refactors, non-trivial bug fixes.

> demo-2

DEMO 2 Plan → Execute → Review

The controller pattern's real value isn't the agents — it's the discipline. Force Claude to write a plan as a document before writing code. You read it. You adjust. Then it builds. Then it reviews itself against the plan.

Why This Wins

1. PLAN → PLAN.md

Claude commits to an approach in writing. You catch wrong assumptions before tokens are spent.

2. EXECUTE

Implement the approved plan. Tests as it goes. One agent or many, in parallel worktrees — doesn't matter, the plan is the spine.

3. REVIEW → REVIEW.md

Diff the result against the plan. Anything skipped, added, or changed gets called out — on the record.

THE INSIGHT A bad plan is cheap to fix. A bad implementation is expensive to throw away.

The Prompt

Use the lowerCaseMe.py from Demo 1. Paste:

Extend lowerCaseMe.py into a multi-mode
text transformer with --upper, --snake,
--kebab, and --leet flags. Add tests.

But work in three steps:

1. PLAN: Write PLAN.md -- the modes,
ordering, file/stdin handling, edge
cases, and the test cases. Stop and
wait for me to approve.

2. EXECUTE: Once I approve, implement
the plan. Run the tests as you go.

3. REVIEW: Write REVIEW.md -- diff the
result against the plan, list any
deviations, and rate your own work.

WATCH FOR Claude stops before code — the plan is the first artifact. You get to steer.

> ~/what-just-happened

What the Plan → Execute → Review Loop Bought You

The Wins

  • You got a PLAN.md to read before any code — the cheap moment to redirect Claude
  • The plan was concrete: file names, functions, edge cases, test cases — not vibes
  • REVIEW.md made deviations visible. Anything Claude skipped or improvised got logged, not buried in the diff
  • The artifacts survive the session. Git-diffable. Reviewable by a teammate.

When to Reach for This

  • Anything you'd hate to throw away — refactors, multi-file changes, non-trivial bug fixes
  • Work where you want a paper trail beyond the chat transcript
  • Tasks big enough that one agent might lose the thread halfway through

Skip It When…

  • The change is one line in one file — planning ceremony adds friction, not value
  • You're exploring — the goal is learning the codebase, not shipping a deliverable
> agent-teams

Agent Teams — Peers Who Disagree EXPERIMENTAL

Multiple Claude sessions in tmux that talk to each other directly — not through you. The unique value isn't parallelism (sub-agents already do that). It's conflict: agents with different roles, different stances, pushing back on each other.

What Makes a Team a Team

  • Conversational — back-and-forth, not request/response
  • Role-oriented — each agent has a stance, not just a task
  • Oppositional — they're allowed to disagree, and the prompt encourages it
  • Autonomous — once you kick them off, they self-coordinate

The thing a single agent can't do for itself: argue with itself convincingly. Two agents with sharp, opposing roles will surface tradeoffs a solo agent waves past.

DEMO 3 The Architecture Debate (team-shaped exercise)

Three agents. Three personalities. One small problem. They argue it out and produce a single recommendation — the disagreement is the feature. Coordination is via shared filesystem (each agent writes to its worktree, reads the others'); see the next slide for what real agent teams add.

The Setup (one command)

$ ./scripts/demo3-debate.sh

The script does everything: three worktrees, a 3-pane tmux session, Claude in each pane with its persona pre-loaded, mouse mode on. Then it auto-fires the kickoff in all three panes after the personas ack.

The Personas (one per pane)

  • MINIMALIST — "the simplest correct implementation always wins. Lines of code are a cost."
  • DEFENSIVE — "handle every edge case up front. Silent bugs are the worst bugs."
  • PERF HAWK — "runtime and memory matter most. Don't ship anything you haven't reasoned about."

CLEANUP ./scripts/demo3-debate.sh --cleanup

The Kickoff (auto-fired by the script)

DEBATE BEGINS. Implement
dedupe(items: list[str]) -> list[str]
case-insensitive, first-seen order.

(1) Write your IMPL.py in your worktree
with role-flavored comments.
(2) Read the others' IMPL.py files.
(3) Write CRITIQUE.md -- your sharpest
objections to each.
(4) Minimalist: write PROPOSAL.md
synthesizing the team. Others:
append agreement or dissent.

WATCH FOR Real pushback in the CRITIQUE files. "I disagree because…" is the moment that justifies the tmux setup.

> ~/honest-disclosure

Honest Disclosure — That Wasn't a Real Agent Team

What you just watched was a team-shaped exercise: three independent Claude sessions coordinating through the filesystem. Real agent teams (the experimental Claude Code feature) add native peer messaging on top of that pattern. Same shape, different plumbing.

What the demo did (filesystem-coordinated)
Three independent claude sessions, no shared channel
Each agent in its own worktree
Coordination by writing & reading files across worktrees
Agents only "hear" each other when they read disk
Works on any Claude Code version — deterministic, demoable
What real agent teams add (experimental)
@-mention a teammate — direct message, not a file
Shared team channel — broadcasts to the group
Recipient sees messages in conversation, mid-turn
More fluid back-and-forth, less polling
Newer, more variable — not what I'd risk on stage yet

The pattern is the same in both: sharp opposing roles, structured disagreement, a converged artifact. The plumbing changes; the value of letting agents argue doesn't.

> ~/which-pattern

Sub-Agents vs Teams vs Single Agent

START Is the task a single, focused change?
  |
  +-- YES → Single Agent (most tasks — don't over-engineer)
  |
  +-- NO → Do you need disagreement, or just parallelism?
       |
       +-- PARALLELISM → Sub-Agents + Plan→Execute→Review
       |
       +-- DISAGREEMENT → Agent Teams (peer mesh in tmux)
          debates, negotiations, adversarial review
       |
       +-- NEITHER → Break it down further, then ask again

DEFAULT When in doubt, single agent. Most tasks don't need parallelism — they need a clearer prompt.

> ~/the-dark-factory

L5 — The Dark Factory

A factory floor that runs without lights, without humans, often through the night. Borrowed by Steve Yegge for the agentic version: no controller, no tmux, no orchestration. You write a prompt; the platform writes the code.

What Changes at L5

  • You stop being the controller
  • The harness picks the agents for you
  • The harness picks the architecture for you
  • You're now the customer, not the engineer
  • Output: a deployable product, not a diff

What You Give Up

  • Legibility — you don't see the team structure
  • Steerability — less mid-flight course correction
  • Cost predictability — tokens go up fast
  • Trust — you have to take the output on faith, or test it

The bargain: trade control for speed. Sometimes that's the right trade. Sometimes it isn't.

The difference from Demo 2: you didn't write the plan. You didn't pick the agents. You didn't create the worktrees. The platform did all of that from a one-paragraph spec.

Bingo in Five Beads

The Mayor read the spec and decomposed it into a five-step convoy. A Polecat built each bead in sequence. The Refinery reviewed the PRs.

2 Seeded RNG + localStorage after 1
3 Click toggle + bingo detection after 1-2
4 Celebration overlay + confetti after 1-3
5 Final polish + responsive after all

Serial pipeline — each bead builds on the previous one's output.

What You're Watching

  • A Polecat picked up each bead, wrote code, ran tests — you didn't prompt it
  • The Refinery reviewed each PR — you didn't ask for a review
  • Every PR that landed created a deployable artifact

The punchline: this is the same Plan→Execute→Review pipeline from Demo 2, but the platform ran the whole thing. You went from controller to customer.

> ~/the-bill

The Bill Comes Due

Token spend looks small until it isn't. As you climb the ladder, the cost shape changes — and "Claude is cheap" stops being true.

L2 — Vibe Coding

~$1-5 per session
One agent, one task, you in the loop

L3-L4 — Orchestration

~$10-50 per feature
N agents, parallel runs, retries

L5 — Dark Factory

~$50-500+ per build
The platform burns aggressively

Saves What

  • Your time — the only non-renewable resource
  • Context-switching cost between tasks
  • "Boring" code (boilerplate, tests, docs)

Spends What

  • Tokens — sometimes 10-100x a single agent
  • Trust — you read fewer lines of what shipped
  • Skill atrophy — junior devs need a path that isn't this

RULE Set a budget alert. Track per-feature cost. The minute orchestration stops paying back, drop a level.

> ~/off-the-beaten-path

Open Harnesses, Open Weights

The bill on the last slide assumed Claude. It doesn't have to. The harness pattern is portable — same agents, same worktrees, same CLAUDE.md-style brief — against open-weight models you can run yourself or rent for cents.

The Open Stack

  • opencode — OSS terminal CLI, similar shape to Claude Code, bring your own model
  • Qwen3-Coder — open-weight from Alibaba; strong on coding + agentic flows; long context
  • Run via ollama / vLLM on your own GPU, or via cheap API providers (OpenRouter, Together, DeepInfra)
# local: zero per-token cost
$ ollama pull qwen2.5-coder:32b
$ opencode --model ollama/qwen2.5-coder:32b

# or via API: pennies per session
$ opencode --model openrouter/qwen3-coder

Why It Matters

  • Cost — pennies per session via API, or zero on your own hardware
  • Sovereignty — code never leaves your machine
  • Tunable — fine-tune the model on your codebase if you want

Honest Tradeoffs

  • Frontier capability still favors closed models — gap is narrowing, not closed
  • Harness ecosystem is younger; rougher edges, fewer skills
  • More setup, more opinion required — you become the platform engineer

The takeaway: the patterns from this hour — sub-agents, controllers, worktrees, harness layers — aren't Claude-specific. They're the shape of agentic coding, full stop. Take them with you, whatever stack you land on.

> ~/appendix/sandbox

APPENDIX
When the Filesystem Isn't Enough

Worktrees protect your repo. They don't protect your machine. For agents that run untrusted code, install random packages, or hit your network — you need a real sandbox.

Sandboxed Bash (Claude Code, macOS)

A built-in seatbelt: bash commands run with restricted file/network access. Cheap, no setup, lives inside Claude Code.

# settings.json
{
  "sandbox": {
    "enabled": true,
    "network": "deny"
  }
}

Best for: running tests, formatters, linters, builds.

Containers (devcontainer / Docker)

Full OS-level isolation. The agent lives in a container that you can nuke and rebuild. Heavier, but the right tool for fully-autonomous work.

# run claude in a fresh devcontainer
$ devcontainer up
$ devcontainer exec claude

Best for: --dangerously-skip-permissions runs, untrusted code, overnight agents.

RULE OF THUMB Worktree for parallelism · sandboxed bash for normal work · container for "I'm not watching this run".
