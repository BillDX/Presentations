> ~/the-ceiling

The Single-Agent Ceiling

One agent. One context window. One task at a time.
As context expands, focus degrades.

# the classic failure mode
$ claude "Refactor auth, update tests, fix the UI"

# step 1: [OK] auth refactored
# step 2: [OK] tests updated
# step 3: [!!] context degraded -- step 1 blurry
# step 4: [XX] /clear and start over
CONTEXT BLOAT

Too many concerns in one window

SEQUENTIAL

Three independent tasks queued single-file

NO SPECIALIZATION

One agent wearing every hat

The fix: decompose, delegate, parallelize.

> sub-agents

Sub-Agents -- The Building Block

Specialized workers spawned within your session. Each gets its own context window, runs independently, and reports results back to the parent.

MAIN SESSION (YOU)

orchestrates + receives results

|--------+--------+--------|
[EXPLORE]

Read-only search

[PLAN]

Architecture

[GENERAL]

Full autonomy

`-- results --> parent <-- results --'

KEY Sub-agents cannot message each other.

THE AGENT DEFINITION FILE

# .claude/agents/reviewer.md
---
name: reviewer
description: Reviews code for
bugs, security, and style.
tools: Read, Grep, Glob
model: sonnet
---
You are a code reviewer. Read the
specified files and report issues
grouped by severity.

KEY LEVERS

  • tools: restrict what the agent can do
  • model: route to opus / sonnet / haiku
  • isolation: worktree for filesystem safety
> exercise-1

EXERCISE 1 Custom Sub-Agents

Your Task

  1. Create .claude/agents/reviewer.md -- a read-only code reviewer
    tools: Read, Grep, Glob | model: sonnet
  2. Create .claude/agents/documenter.md -- generates docstrings
    tools: Read, Write, Edit | model: haiku
  3. Open Claude and prompt it to use both agents on a file in your project
  4. Observe: agents run in parallel, each with isolated context

Example Prompt

$ claude

Use two sub-agents on src/api.py:

1. The reviewer agent: find bugs,
security issues, and style problems
2. The documenter agent: add docstrings
to all public functions

Run both agents in parallel.

Bonus: Add a third agent -- a security-reviewer that checks for OWASP top 10 issues.

15 MINUTES Go! Ask me if you get stuck.

> debrief-1

Debrief: Sub-Agents

What Just Happened

  • Each agent got a fresh context window -- no crosstalk
  • The reviewer couldn't write files -- tool restrictions enforced
  • The documenter used a cheaper model -- cost control
  • Both ran in parallel -- faster than sequential

Common Gotchas

  • Agents can't talk to each other -- only back to parent
  • Too-vague system prompts = generic results
  • Forgetting description: field -- Claude needs it to know when to use the agent

Key Insight

Agent definitions are reusable. Build a library of specialists. Ship them in your repo's .claude/agents/ directory.

> controller-pattern

The Controller Pattern

A single orchestrator session spawns specialized sub-agents in worktrees. You stay in control. Each agent works in isolation and reports back.

THE PIPELINE

CONTROLLER (YOU)
|--- spawns ---|
1. PLANNER

Read spec → plan

2. CODER

Plan → code

3. TESTER

Code → tests

`-- results flow back to controller --'

WHY WORKTREES?

  • Each agent gets its own filesystem copy
  • No file collisions -- coder and tester can't step on each other
  • Results merge back via git
  • If an agent breaks something, only its worktree is affected
# worktree = separate checkout
$ git worktree list
/project abc1234 [main]
/project-coder def5678 [coder-branch]
/project-tester ghi9012 [tester-branch]
> exercise-2

EXERCISE 2 The Controller Pipeline

Your Task

  1. Open your project in Claude with --worktree controller
  2. Create a planner agent (Read, Grep, Glob | model: opus)
  3. Create a coder agent (Read, Write, Edit, Bash | isolation: worktree)
  4. Create a tester agent (Read, Write, Edit, Bash | isolation: worktree)
  5. Prompt the controller to run the pipeline on SPEC.md

Example Prompt

Implement the feature in SPEC.md.
Use this pipeline:

1. planner agent reads SPEC.md and
produces a step-by-step plan
2. coder agent implements the plan
in a worktree
3. tester agent writes and runs tests
in a separate worktree

Run coder and tester in parallel
after the plan is ready.

TIP Check git worktree list to see the isolation in action.

15 MINUTES Go! Watch the worktrees appear.

> debrief-2

Debrief: Controller Pattern

What You Built

  • A planning phase that produced structured output
  • Parallel execution -- coder and tester working simultaneously
  • Worktree isolation -- no file conflicts between agents
  • The controller orchestrated the flow and collected results

When to Use This

  • Feature implementation from a spec
  • Multi-service changes (frontend + backend + tests)
  • Code review pipelines (lint + security + test)
  • Any workflow with clear sequential + parallel phases

When NOT to Use This

  • Simple single-file changes
  • Tasks that need agents to talk to each other

Agent Teams -- Peer Collaboration EXPERIMENTAL

Multiple Claude Code sessions that coordinate as peers in tmux. A lead initiates the team, but all agents message each other directly.

[LEAD]

Creates + assigns

CODER

Implements

REVIEWER

Reviews

TESTER

Validates

SUB-AGENTS vs AGENT TEAMS

Sub-Agents
Hub-and-spoke

Cheaper, simpler
Agent Teams
Peer-to-peer mesh

More flexible, experimental

Part A: Watch the Demo

> decision-tree

Choosing Your Weapon

START Is the task a single, focused change?
  |
  +-- YES → Single Agent (most tasks)
  |
  +-- NO → Can you decompose into independent sub-tasks?
       |
       +-- YES → Do tasks need to talk to each other?
       |       |
       |       +-- NO → Sub-Agents (hub-and-spoke)
       |       +-- YES → Agent Teams (peer mesh)
       |
       +-- NO → Break it down more, then try again
Sub-Agents Best For

Pipelines, reviews, parallel tasks that report to a controller

Agent Teams Best For

Complex builds, TDD loops, tasks requiring negotiation between agents

Worktrees Best For

Any multi-agent scenario where agents write code to the same repo
