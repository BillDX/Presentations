> ~/codebase

The Single-Agent Ceiling

One agent. One context window. One task at a time.
As context expands, focus degrades.

# the classic failure mode
$ claude "Refactor auth across 3 services, update tests, fix the UI"

# step 1: [OK] auth service refactored
# step 2: [OK] user service updated
# step 3: [!!] context degraded -- details from step 1 blurry
# step 4: [XX] /clear and start over
CONTEXT BLOAT

PM notes + CSS bugs + API schemas = unfocused agent

SEQUENTIAL BOTTLENECK

Three independent tasks queued single-file

NO SPECIALIZATION

One agent wearing every hat -- security, perf, UX, tests

The fix: decompose, delegate, parallelize.

> ~/the-real-cost

The Core Bottleneck: Time

A single agent is cheap in Claude tokens.
It is extremely expensive in the only non-renewable resource you have.

LOW

Claude Tokens

Single context window.
Minimal parallel spend.
Looks cheap on the bill.

MAXIMUM

Time and Attention

Watching. Waiting. Approving.
Context-switching. Re-explaining.

The goal of orchestration is not to save tokens.

It is to reclaim your time -- delegate, parallelize, and let agents work while you do something else. Spend more Claude tokens. Spend fewer you tokens.

> sub-agents

Sub-Agents -- Task Oriented

Specialized workers spawned within your session. Each gets its own context window, runs independently, and reports results back to the parent.

MAIN SESSION (YOU)

orchestrates + receives results

|----------+-----------+----------|
[EXPLORE] agent

Read-only. Searches and analyzes the codebase.

[PLAN] agent

Designs architecture and implementation strategy.

[GENERAL] agent

Custom prompt, tool access, full autonomy.

`--- results -->> parent <<-- results ---'

KEY Sub-agents cannot message each other -- they only report back to the parent. The main agent is the sole coordinator.

Defining Custom Sub-Agents

MARKDOWN + YAML FRONTMATTER

# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for
security vulnerabilities
tools: Read, Grep, Glob
model: sonnet
isolation: worktree
---

You are a security specialist.
Focus on: injection, auth bypass,
secrets exposure, SSRF, XSS...

WHY SUB-AGENTS?

  • Context isolation -- exploration stays out of your main conversation
  • Tool restrictions -- read-only agents cannot break things
  • Cost control -- route to Haiku for cheap tasks
  • Parallel execution -- up to ~7 simultaneous tasks
  • Reusability -- ~/.claude/agents/ works across all projects
# invoke explicitly
"Use the security-reviewer
agent on src/auth/"

# or Claude auto-delegates
# based on description match
> sub-agents in action

Sub-Agent Examples

Define the agents with .md files in the .claude/agents directory, or bring them into being on the prompt line.

THE AGENTS

# .claude/agents/planner.md --- name: planner description: Analyzes specs and produces an implementation plan. Never writes code. tools: Read, Grep, Glob model: opus --- Read the spec and codebase. Produce a step-by-step implementation plan with file paths and function names.
# .claude/agents/coder.md --- name: coder description: Implements features from a plan. Commits to branch. tools: Read, Write, Edit, Bash, Grep model: opus isolation: worktree --- Implement exactly what the plan says. Commit after each logical unit.
# .claude/agents/tester.md --- name: tester description: Writes and runs tests against new code. tools: Read, Write, Edit, Bash, Grep model: sonnet isolation: worktree --- Write tests for the new code. Run the full suite. Report pass/fail with exact output.
# .claude/agents/merger.md --- name: merger description: Merges worktree branches and opens PRs. tools: Read, Write, Edit, Bash, Grep --- Merge branches into main. Resolve conflicts. Run tests. Open a PR.

THE PROMPT

$ claude "Build the payment integration from SPEC.md. Use this pipeline: 1. planner agent reads the spec 2. coder agent implements the plan 3. tester agent validates the code 4. merger agent opens a PR Run steps 2+3 in parallel with worktree isolation."

NOTE The parent session stays in control. Each agent returns a summary -- planner returns the plan, coder returns a diff, tester returns pass/fail, merger returns a PR link. No agent talks to another directly.

> sub-agent-controller

Step by Step: Sub-Agent Controller

A single orchestrator session spawns specialized sub-agents in worktrees. You stay in control. Each agent works in isolation and reports back.

1. START THE CONTROLLER

# start a tmux session
$ tmux new -s team

$ cd ~/your-project

# launch Claude in its own worktree
# as the controller session
$ claude \
    --worktree controller \
    --dangerously-skip-permissions

2. SPAWN THE TEAM VIA PROMPT

Use sub-agents with worktree isolation. Spawn four sub-agents: 1. merge-manager Handles git operations, rebasing, conflict resolution. Merges completed worktree branches back to main. Opens PRs. 2. frontEnd-coder Implements the FastAPI front end. Templates, routes, static assets. 3. backEnd-coder Implements back end features. Models, services, API endpoints. 4. test-bot Implements Red/Green TDD tests. Writes failing tests first, then validates implementations pass. Use worktrees for your agents. Coordinate their work and merge results.

WHY THIS PATTERN?

• Controller worktree -- no conflicts • Agent worktrees -- no file collisions • Merge-manager handles git • Results flow to controller

KEY DIFFERENCE FROM TEAMS These four agents cannot message each other. The controller orchestrates everything: dispatches work, collects results, decides when to call merge-manager. Simpler, cheaper, and you keep full control of the flow.

Agent Teams -- Collaboration EXPERIMENTAL

Multiple Claude Code sessions that coordinate as peers. A lead initiates the team, but all agents message each other directly.

Creates team. Assigns tasks. Synthesizes results.

> tmux + agents

Hands-On: Agent Team in tmux

Four specialized agents, each in its own pane. The controller orchestrates, the coder builds, the tester validates, and the merger handles PRs.

1. DEFINE YOUR AGENTS

# .claude/agents/controller.md --- name: controller description: Orchestrates the team. Breaks tasks into units. Assigns work. Reviews output. Never codes. tools: Read, Grep, Glob, Task --- You are the project lead. Decompose work into discrete tasks. Delegate to coder and tester agents. Review all output before approving merge.
# .claude/agents/coder.md --- name: coder description: Writes implementation code. Follows specs exactly. tools: Read, Write, Edit, Bash, Grep model: opus isolation: worktree --- You implement features. Write clean, tested code. Commit frequently with descriptive messages.
# .claude/agents/tester.md --- name: tester description: Writes and runs tests. Validates code from the coder. tools: Read, Write, Edit, Bash, Grep model: sonnet isolation: worktree --- You write tests and validate work. Run the full suite after each change. Report failures with exact output.
# .claude/agents/merger.md --- name: merger description: Merges worktree branches. Resolves conflicts. Opens pull requests. tools: Read, Write, Edit, Bash, Grep --- You handle git operations. Merge worktree branches into main. Resolve conflicts. Run tests post-merge. Open a PR with a summary of changes.

2. LAUNCH IN TMUX

$ tmux new -s build-team && claude "Create an agent team. Use the controller, coder, tester, and merger agents. Build the payment integration feature from SPEC.md."
> tdd-team-walkthrough

Step by Step: TDD Team in tmux

1. START THE LEAD SESSION

# start or attach a tmux session
$ tmux new -s team

$ cd ~/your-project

# launch Claude
# agent teams auto-detect tmux
# and use split panes
# all teammates inherit the lead's
# permission settings
$ claude --dangerously-skip-permissions

2. SPAWN THE TEAM VIA PROMPT

Create an agent team with split-pane display. Spawn 3 teammates: 1. Red Test Writer Writes failing tests first (TDD red phase). Focus on edge cases, error paths, and boundary conditions. Use worktree isolation. 2. Green Phase Implementer Takes Red's failing tests and writes the minimal implementation to make them pass (TDD green phase). Use worktree isolation. 3. Senior Engineer Reviewer Reviews both the tests and implementation. Checks for code quality, architecture, security, and coverage gaps. Require plan approval before suggesting refactors. Enable delegate mode so you only coordinate -- don't write code yourself. Use the shared task list to track Red >> Green >> Review dependencies.

WHY THIS WORKS Red and Green message each other directly -- Red sends failing tests, Green gets them in real time. The Reviewer watches both and can challenge decisions before code is merged. The shared task list enforces the Red >> Green >> Review dependency chain.

> agent-teams

Under the Hood

COORDINATION PRIMITIVES

[TASK LIST]

Shared work items with dependency tracking. Tasks auto-unblock when dependencies complete.

[MAILBOX]

Direct messaging between agents -- not just report-to-lead. Teammates debate, challenge, share discoveries mid-task.

[LIFECYCLE]

Lead spawns >> teammates work >> synthesize >> cleanup. Built into Claude Code core.

DISPLAY MODES

# three display modes: in-process (default) # all in one terminal # Shift+Up/Down to switch split-panes (recommended) # each teammate = own tmux pane # watch them all at once auto # tmux detected? >> split panes # otherwise >> in-process

TMUX SETUP

# start a tmux session first $ tmux new -s team-review # launch claude inside -- auto splits $ claude "Create an agent team to review..."

COST Each teammate is a full context window with its own CLAUDE.md read, codebase indexing, and message history. 3 teammates = 3 complete sessions running in parallel. ~5x tokens. Agents can be chatty.

> diff

Sub-Agents vs Agent Teams

[SUB-AGENTS]
Own context, results return to caller
Report back to main agent only
Main agent manages everything
Lower token cost -- results summarized
Best for focused, scoped tasks
Up to ~7 parallel tasks
Stable -- built-in feature
[AGENT TEAMS]
Own context, fully independent sessions
Teammates message each other directly
Shared task list with self-coordination
Higher token cost -- full context per agent
Best for work needing discussion/debate
Scales to team-sized workloads
Experimental -- env flag required

RULE OF THUMB

Workers only need to return results?  -->>  Sub-agents. Cheaper, simpler, proven.

Workers need to talk to each other?  -->>  Agent teams. Shared task list + mailboxes.

> isolation

Git Worktrees -- Filesystem Isolation

Multiple agents editing the same files = disaster. Worktrees give each agent its own checkout, branch, and working directory -- sharing the same repo.

CLI USAGE

# named worktree $ claude --worktree feature-auth
# second session, fully isolated $ claude --worktree bugfix-123
# combine with tmux $ claude --worktree feat --tmux
[MERGE AGENT]

When worktree work is done, use a sub-agent to handle the merge:

"Spawn a sub-agent to merge the feature-auth worktree into main. Resolve any conflicts, run tests, and open a PR."

IN SUB-AGENT FRONTMATTER

--- name: migrator isolation: worktree # <-- this --- Each invocation gets its own worktree. Auto-cleanup on exit.

PROMPTING WORKTREES

# ask Claude directly in-session "Use worktrees for your agents." # or be explicit about the split "Spawn 3 sub-agents with worktree isolation. One for API, one for UI, one for tests."

TIP Add .claude/worktrees/ to .gitignore

> examples

[BUG TRIBUNAL] AGENT TEAMS

Five agents investigate a bug. They debate competing theories and challenge each other. The team lead synthesizes the verdict.

Create an agent team to investigate the WebSocket disconnect bug.
Spawn five investigators:
- One checking the connection lifecycle
- One analyzing the nginx proxy config
- One reviewing recent deployment diffs
- One profiling memory and event loop under load
- One playing devil's advocate -- challenging every theory

Have them share findings and debate before converging on a root cause.

Why teams? The devil's advocate needs to read others' findings in real time. Sub-agents would investigate in isolation and never challenge each other.

> examples

[FULL-STACK BUILD] SUB-AGENTS + WORKTREES

Three sub-agents in isolated worktrees, each owning a layer. The parent merges when all report back.

[API] agent

isolation: worktree
Writes endpoints, migrations, validations on its own branch.

[UI] agent

isolation: worktree
Builds React components, hooks, styles on a separate branch.

[TEST] agent

isolation: worktree
Writes integration + unit tests against both layers.

`--- all report back --- merge --- [OK] ---'
# why sub-agents here (not teams)?
# these three do not need to talk to each other.
# API types are shared via committed code, not chat.
# lower token cost. worktrees prevent file conflicts.

[QA SWARM] AGENT TEAMS

Point a team at your running app. Each teammate tests through a different lens. They find things a single reviewer always misses.

Use a team of agents to do QA against my app.
It is running at http://localhost:3000/

Spawn teammates:
- Accessibility auditor (WCAG 2.1 AA)
- Security pen tester (OWASP top 10)
- Performance profiler (Core Web Vitals)
- UX critic (mobile-first, dark mode, edge cases)

Have them coordinate -- if the security tester finds an endpoint,
the performance tester should load-test it too.

Why teams? Security discovers an unauth endpoint >> messages perf to stress it >> messages a11y to check its error page. Real-time coordination.

> sandboxing

Built-in Sandbox -- /sandbox

Claude Code ships with OS-level sandboxing. On macOS it uses Apple Seatbelt (sandbox-exec). It enforces two boundaries: filesystem isolation and network isolation.

HOW IT WORKS

[FILESYSTEM]

Read/write to CWD and subdirectories. Read-only to the rest of the system. Sensitive dirs blocked entirely. Configurable allow/deny paths.

[NETWORK]

Proxy server runs outside the sandbox. Only approved domains are reachable. Blocks exfiltration of SSH keys, tokens, secrets.

[CHILD PROCESSES]

All spawned scripts, subprocesses, and sub-agents inherit the same sandbox boundaries. No escape via fork.

SETUP

# enable from inside Claude Code $ claude /sandbox # choose a mode: auto-allow # sandboxed cmds run without prompts # unsandboxable cmds fall back to # normal permission flow permissioned # sandboxed cmds still ask permission # but get sandbox enforcement on top

RESULT Anthropic reports 84% fewer permission prompts with sandboxing enabled internally.

> macos-isolation

macOS Isolation Methods

Beyond the built-in sandbox, several approaches provide harder boundaries -- useful when running agents unattended or with --dangerously-skip-permissions.

[DOCKER SANDBOXES] RECOMMENDED

Docker Desktop 4.58+ runs each agent in a dedicated microVM. Network allow/deny lists, Docker-in-Docker support, instant reset. Works with Claude Code, Codex, Gemini CLI, Kiro.

$ docker sandbox run -v ./:/workspace -- claude --dangerously-skip-permissions
[APPLE CONTAINER] macOS 26+

Apple's own lightweight Linux VM tool. Sub-second startup on Apple Silicon. No daemon, no license. Bind-mount your project in, everything else stays out.

$ container run -it --rm -v "$PWD:/workspace" claude-sandbox
[LUME / TART VM]

Full macOS VM via Apple Virtualization.framework. Claude Code on your host SSHs into the VM. Strongest isolation -- complete OS boundary. Share project via shared folders or git push.

$ lume run sandbox --shared-dir "$PWD" "SSH into lume@192.168.64.X and work on /Volumes/My Shared Files"
[CUSTOM SEATBELT]

Roll your own sandbox-exec profile. Tools like claude-sandbox wrap Claude with a Seatbelt policy: deny all writes, allow CWD + ~/.claude + /tmp. Lightweight, no VM overhead.

$ claude-sandbox claude # sandbox-exec with custom .sb profile
> choosing-isolation

The Isolation Spectrum

LIGHTEST

Built-in /sandbox

Seatbelt + proxy
Zero setup
~0s overhead

LIGHT

Custom Seatbelt

Stricter read policy
No VM needed
~0s overhead

MEDIUM

Docker Sandboxes

MicroVM per agent
Docker-in-Docker
~2-5s startup

HEAVY

Full macOS VM

Lume / Tart
Complete OS boundary
~10-30s startup

WHEN TO USE WHAT

Built-in /sandbox -- interactive daily work. You are present and reviewing output.

Custom Seatbelt -- tighter read restrictions when working near sensitive dirs.

Docker Sandboxes -- unattended agents, CI, or --dangerously-skip-permissions runs.

Full VM -- untrusted code, production-adjacent experiments, maximum blast-radius control.

Key insight: effective sandboxing needs both filesystem and network isolation. Without network isolation, a compromised agent exfiltrates your keys. Without filesystem isolation, it escapes to gain network access.
